摘要

1. 引言 视觉语言模型的兴起开启了计算机使用智能体的新时代，使其能够与图形用户界面（GUI）交互以执行复杂任务（Wang et al., 2025a; Qin et al., 2025; He et al., 2024; Zhang et al., 2025）。尽管进展迅速，一个根本性瓶颈依然存在：定位问题（grounding problem）（Cheng et al., 2024; Feizi et al., 2025）。为了有效运行，屏幕智能体必须首先准确识别 UI 元素，理解它们的角色，并推理它们的空间和功能关系。这种结构性理解是有效进行下游规划和动作执行的先决条件；当它失败时，错误会在整个智能体流水线中级联传播。

现代计算机使用智能体（computer-use agents, CUA）必须将屏幕感知为一种结构化状态——哪些元素可见、它们位于何处、以及它们包含什么文本——然后才能可靠地定位指令并采取行动。然而，大多数现有定位数据集提供的是稀疏监督，其标签不足且多样性低，每个屏幕只标注一小部分与任务相关的元素，这限制了覆盖范围和泛化能力；此外，实际部署需要高效率，以支持低延迟、端侧使用。我们提出 ScreenParse，这是一个用于完整屏幕解析的大规模数据集，包含对 771K 张网页截图中所有可见 UI 元素（框、55 类类型和文本）的密集标注（21M 个元素）。ScreenParse 由 Webshot 生成，Webshot 是一个自动化、可扩展的流水线，可渲染多样化的 url，提取标注，并应用基于 VLM 的重新标注和质量过滤。利用 ScreenParse，我们训练了 ScreenVLM，这是一个紧凑的、316M 参数的视觉语言模型（vision language model, VLM），它解码一种紧凑的 ScreenTag 标记表示，并采用一种结构感知损失，对结构关键 token 赋予更高权重。ScreenVLM 在密集解析上显著优于规模大得多的基础 VLM（例如，在 ScreenParse 上 PageIoU 为 0.592，而对比为 0.294），并在公开基准上表现出很强的迁移能力。此外，在 ScreenParse 上微调基础 VLM 能够持续提升其定位性能，这表明密集屏幕监督为 UI 理解提供了可迁移的结构先验。项目页面：https://saidgurbuz.github.io/screenparse/。

当前用于 GUI 交互的最先进模型主要依赖“稀疏”的面向动作的数据集，这些数据集只标注每个任务步骤中相关的单个元素（Deng et al., 2023; Zhou et al., 2024; Rawles et al., 2023; Xie et al., 2024）。这种监督对于端到端策略是有价值的，但它使屏幕上大多数元素未被标注，并使完整屏幕结构保持隐式。因此，模型可能学到足以应对受监督步骤的捷径，却未能形成完整的屏幕状态，这可能损害其对新布局、新应用以及分布外屏幕的鲁棒性和泛化能力。此外，实际部署通常需要低延迟、端侧推理，这促使人们采用紧凑的感知模型，而不是完全依赖大型基础 VLM（Bai et al., 2025; Zhu et al., 2025）。我们认为，一种自然的补救方法是将完整屏幕解析作为核心训练目标。我们将屏幕解析定义为恢复屏幕的完整语义结构：所有可见 UI 元素的集合、它们的边界框、语义类型以及相关文本。与单目标定位相比，密集屏幕解析提供了整体性的屏幕理解，下游智能体可以在此基础上进行指令遵循和动作选择。

UI 定位基准和数据集。一个密切相关的研究方向是 UI 定位，其中模型定位由自然语言指令所指称的元素。SeeClick 和 ScreenSpot/ScreenSpotPro 推广了以指令为条件的定位评估，而一项同期工作 GroundCUA 提供了由人类演示派生的更完整的屏幕级标注（Cheng et al., 2024; Li et al., 2025; Feizi et al., 2025）。然而，大多数基准提供稀疏监督（一条指令对应一个或少数几个元素），而像 GroundCUA 这样更完整的数据集在规模和多样性方面受到限制。相比之下，我们的数据集面向完整屏幕解析，提供渲染后几乎所有可见 UI 元素的密集标注，从而提供一种整体性的感知先验，补充指令级定位。

UI 标注，然后应用基于视觉语言模型（VLM）的精炼与质量过滤，以生成高质量数据集。图 3 总结了 Webshot 流水线。借助我们的 Webshot 流水线，我们构建了 ScreenParse，这是一个用于完整屏幕解析的大规模数据集，为所有可见 UI 元素提供密集标注，包括其边界框、语义类型和文本，覆盖 55 个 UI 类别。基于这些数据，我们训练了 ScreenVLM，这是一种超轻量级视觉–语言模型，可将完整屏幕解析为一种结构化序列表示 ScreenTag。为了使优化更好地与结构化抽取对齐，我们进一步引入了一种结构感知加权损失，对结构关键 token（例如标签和位置）赋予更高权重，从而提升预测布局的保真度。实证结果表明，ScreenVLM 在密集解析方面显著优于规模大得多的基础 VLM 基线，并能有效迁移到公开基准。此外，我们证明 ScreenParse 监督也有益于其他模型家族，同时增强了基础 VLM 和基于检测器的解析器。这些结果表明，密集屏幕监督为鲁棒 UI 理解提供了可迁移的结构先验。

基础 VLM 与解析器。基础 VLM，如 Qwen3-VL、InternVL3 和 Gemini-2.5，可通过提示用于各种下游应用（Yoon et al., 2025; Han et al., 2025）。除它们之外，定位和结构化抽取是 GUI 感知的常见基线（Bai et al., 2025; Zhu et al., 2025; Comanici et al., 2025）。与此同时，OmniParser 等专门解析器通过基于检测器的流水线定位 UI 元素（Lu et al., 2024）。在实践中，基础 VLM 对于低延迟或端侧部署而言通常过大，而检测器式解析器侧重于定位，缺乏面向下游推理的语言扎根式结构化理解。我们的工作通过使用密集监督来训练用于完整屏幕解析的超紧凑 VLM，弥合了这一差距；并且我们进一步表明，相同的监督在我们的基准和公开评测上都能改进基础 VLM 与基于检测器的解析器（Feizi et al., 2025; Cheng et al., 2024）。

总之，我们的主要贡献如下：• 我们提出 ScreenParse，这是一个用于完整屏幕解析的大规模数据集，提供所有可见 UI 元素的密集标注，例如边界框、元素类型和文本，覆盖 55 个 UI 类别。• 我们提出 Webshot，这是一条可扩展且完全自动化的流水线，可从渲染网页中收集密集的、保持层级结构的屏幕解析标注。

3. 数据集：ScreenParse

• 我们表明，在 ScreenParse 上训练能够带来强大且可迁移的增益：我们提出的 ScreenVLM 架构，以及现有基础 VLM 和最先进的基于检测器的解析器，在我们的基准和公开 UI 理解基准上均有显著提升。

3.1. 概述 本节介绍 ScreenParse，这是一个用于完整屏幕解析的网页规模数据集；完整屏幕解析任务旨在恢复屏幕上所有已渲染的可见 UI 元素及其位置、语义类型和文本。不同于大多数 GUI 智能体与定位数据集仅为被交互或指令所指涉的元素提供稀疏监督（Deng et al., 2023; Zhou et al., 2024; Rawles et al., 2023; Cheng et al., 2024; Xie et al., 2024），ScreenParse 提供密集的屏幕级标注，鼓励整体屏幕理解，并使训练能够跨多样布局泛化的解析器成为可能。

2. 相关工作 计算机使用智能体与评测。近期基准评估能够感知屏幕并在 Web 和 OS 环境中执行动作的端到端智能体，涵盖交互式任务和基于演示的设置（Zhou et al., 2024; Koh et al., 2024; He et al., 2024; Xie et al., 2024; Deng et al., 2023）。更新近的套件（Wang et al., 2025b）进一步将评测扩展到结构化、多平台协议。尽管这些基准对于评估长程成功至关重要，但它们往往将细粒度感知隐含化；我们的工作通过为训练和评测启用密集屏幕级监督来针对这一缺口。

4. 方法 4.1. 问题形式化 给定一张截图 I ∈ RH×W ×3，屏幕解析旨在恢复可见 UI 元素的完整集合，以及它们的几何信息、语义类型和文本内容。具体而言，我们将一个屏幕表示为元素集合 S = {ei }N i=1，其中每个元素 ei = (bi , ci , ti ) 由边界框 bi = (x1 , y1 , x2 , y2 )、来自固定 UI 分类体系的类别标签 ci，以及可选的可见文本 ti 组成。不同于单目标 grounding，其目标是预测屏幕上的所有元素，包括细粒度组件和具有语义意义的容器，从而实现整体屏幕理解。

ScreenVLM。ScreenVLM 是一种紧凑的视觉–语言模型，可将截图转换为序列化的结构化屏幕表示（ScreenTag；图 4）。我们没有引入沉重的新架构，而是将一种 document-to-markup VLM，即 Granite Docling，适配到 UI 领域；该模型基于 Idefics3 系列（Laurençon et al., 2024），并与 SmolDocling（Nassar et al., 2025）密切相关。具体而言，ScreenVLM 将一个强大但高效的视觉骨干网络（Tschannen et al., 2025）与轻量级 Granite 165M 自回归解码器（Mishra et al., 2024）相结合，前者专门用于多模态下游任务（Shin et al., 2024；Cho et al., 2024；Kim et al., 2025）。图像被编码为一小组视觉 token，这些 token 对解码器进行条件化，使其通过标准自回归训练生成 ScreenTag 序列。我们从预训练的 Granite Docling 检查点初始化，因为其文档转换预训练通过类标记输出强调具有定位感知的结构化抽取，这种归纳偏置可以自然迁移到完整屏幕解析。

VLM 展现出很强的解析性能，并超过了规模大得多的模型。特别是，与 Qwen3-VL8B 相比，ScreenVLM 将 PageIoU 从 0.294 提升到 0.606，并达到 Label PageIoU 0.197，表明其具有更好的全局覆盖能力和更准确的标签感知结构恢复能力。InternVL3-2B 和 Qwen3-VL-2B 的表现显著更低（PageIoU 分别为 0.111 和 0.228）。总体而言，这些结果凸显了 ScreenParse 作为密集监督的价值：提示式 VLM 往往会漏掉许多 UI 元素，而使用完整标注进行训练则能显著更强地恢复屏幕结构。

提升了迁移能力：InternVL3-2B 从 0.025/0.006 提升到 0.203/0.036，缩小了与 ScreenVLM 的差距，并支持这样一种观点：密集屏幕解析监督具有广泛益处，而非仅适用于特定模型。基于检测器的解析器在 GroundCUA 上获得了更高的绝对定位分数（例如，微调后的 OmniParser v2 达到 PageIoU 0.398），这与其专业化特性以及该评估对框重叠的强调相一致。重要的是，ScreenVLM 缩小了与检测器流水线之间的差距，同时生成结构化、语言兼容的输出，可被下游智能体直接使用。

在 ScreenParse 上训练的基于检测的模型同样表现强劲。RT-DETRv2 达到 PageIoU 0.600 和 mAP@50 0.362，而 YOLO 达到 PageIoU 0.533 和 mAP@50 0.299；考虑到它们以检测为中心的架构，这是符合预期的。然而，检测器不会产生适用于下游指令条件推理或动作生成的语言接地屏幕状态。相比之下，ScreenVLM 输出一种结构化的、与语言对齐的表示，统一了几何、语义和文本，使其更适合作为预训练模型，并被微调用于计算机使用任务。因此，我们分别报告 VLM 和检测器，并根据它们各自的优势来解释结果。

ScreenSpot：稀疏标注下的接地式评估。表 5 报告了在 ScreenSpot（Web/PC/Mobile）上的结果。由于 ScreenSpot 仅提供稀疏标注，PageIoU 等完整布局指标无法直接适用；因此我们报告 Recall@50 和 PixCov。在 Web 划分上，ScreenVLM 达到 Recall@50 0.557 和 PixCov 0.746，而检测器式解析器取得了更高的召回率（例如，RT-DETRv2 Recall@50 0.768，PixCov 0.857）。在 PC 和 Mobile 上，ScreenVLM 的 Recall@50 分别下降到 0.222 和 0.066，但 PixCov 仍保持在较高水平，分别为 0.839 和 0.847。这种差异表明，在分布外 UI 风格下，ScreenVLM 经常预测出覆盖标注像素的区域（高 PixCov），但未能产生 Recall@50 所要求的紧密元素级框，而 Recall@50 对精确定位更加敏感。总体而言，结果表明 ScreenParse 监督具有非平凡的迁移能力，同时也揭示了一个明确的局限和未来方向：将密集监督扩展到网页之外，以更好地匹配 PC/Mobile UI 分布并提升元素级召回率。

迁移到 GroundCUA。表 4 评估了向 GroundCUA 的迁移，该数据集包含来自多种应用程序的 UI 截图，相对于我们仅基于 Web 的 ScreenParse 训练而言是分布外的。尽管存在这种分布转移，ScreenVLM 仍然是最强的 VLM 基线，达到 PageIoU 0.251 和 Label PageIoU 0.043，而 Qwen3-VL8B 为 0.060/0.010，InternVL3-2B 为 0.025/0.006。这表明密集解析监督会诱导出能够迁移到 Web 领域之外的结构先验。

6. 结论 我们提出了 ScreenParse，这是一个用于完整屏幕解析的 Web 规模数据集，包含由我们的自动化 Webshot 流水线生成的密集 UI 元素标注（框、55 类类型和文本）。利用 ScreenParse，我们训练了 ScreenVLM，这是一个紧凑型 VLM，可预测结构化屏幕状态；并引入了一个结构感知目标，用于强调结构关键 token。在 ScreenParse 上的域内评估以及向公共基准（GroundCUA 和 ScreenSpot）的迁移中（Feizi et al., 2025; Cheng et al., 2024），ScreenVLM 大幅优于提示式基础 VLM 基线；同时，在 ScreenParse 上微调多个基础 VLM 和解析器都带来一致改进，支持了密集屏幕级监督对于整体 UI 理解的价值。

同样的趋势也适用于不同的模型家族：微调后，InternVL3-2B 在 ScreenParse 上的 PageIoU 从 0.111→0.509，Label PageIoU 从 0.036→0.174；在 GroundCUA 上的 PageIoU 从 0.025→0.203，Label PageIoU 从 0.006→0.036。综合来看，这些结果表明 ScreenParse 的收益并非特定于某个模型：密集的屏幕级监督能够持续增强整体屏幕理解并迁移到新的 UI 领域，有时甚至使更小的微调基础模型优于更大的提示式对应模型。

局限性。虽然我们表明，在 ScreenParse 上训练的模型能够较为合理地迁移到其他领域，但 ScreenParse 主要以 Web 为中心，不可避免地与原生桌面/移动应用程序和 UI 工具包之间存在领域差距。此外，尽管 Webshot 进行了大量过滤和精炼，基于 DOM 的提取仍可能包含来自动态内容（例如广告、覆盖层）和渲染伪影的残余噪声，这可能会影响一部分标注。

7.2. 训练细节

Qwen3-VL-2B-Instruct 微调。我们在 ScreenParse 上使用 BF16 和 DeepSpeed ZeRO-3 offload 对 Qwen3-VL-2B-Instruct 进行微调，仅更新多模态 LLM（冻结视觉塔和投影器）。我们使用批大小 1，梯度累积 4，采用 AdamW 优化器和余弦调度（3% warmup），学习率 2 × 10−5，权重衰减 0.01，最大序列长度 8192，并训练 5 个 epoch。

InternVL3-2B 微调。我们使用 BF16 和 DeepSpeed（ZeRO stage 1）在 ScreenParse 上微调 InternVL3-2B，冻结视觉骨干网络，同时更新 LLM 和 MLP。我们训练 5 个 epoch，总批大小为 128（8 个 GPU，每个 GPU 批大小 4，梯度累积 4），使用 AdamW 和余弦调度（3% warmup），学习率 2 × 10−5，权重衰减 0.05，最大序列长度 8192，并使用梯度检查点。

RT-DETRv2 训练。我们在 COCO 格式数据集（55 个类别）上训练 RT-DETRv2，其包含 PResNet-50-VD 骨干网络、HybridEncoder 和 RTDETRTransformerv2。训练使用总批大小 128（验证为 64），图像调整为 736 × 1280，并采用包括 photometric distort、zoom-out、IoU crop 和 random horizontal flip 在内的数据增强；在第 71 个 epoch 之后禁用重度增强和多尺度。我们使用 AdamW 进行优化（lr 8 × 10−4，betas 0.9/0.999，权重衰减 1 × 10−4），对骨干网络使用较低的 LR（8 × 10−5），并对编码器/解码器的 norm/bn 使用零权重衰减。我们训练 72 个 epoch，前 2000 次迭代进行线性 warmup，并使用 MultiStepLR（milestone 1000，gamma 0.1），同时将梯度裁剪到 0.1。

YOLOv11-L 训练。我们在 ScreenParse 上以 1280 分辨率训练 YOLOv11-L 500 个 epoch，在 8 个 GPU 上批大小为 48。我们使用 AdamW 和余弦学习率调度（lr0 = 2.08 × 10−4，lrf = 0.05），momentum 0.9，权重衰减 5 × 10−4，并采用较短的 warmup（≈0.5 个 epoch）和 25 个 epoch 的 patience。我们禁用重度组合增强（mosaic/mixup），转而使用轻度几何/颜色抖动以及多尺度训练。

OmniParser v2 微调。我们使用与 YOLOv11-L 训练相同的训练设置和增强配置来微调 OmniParser v2，仅更改优化调度：我们训练 100 个 epoch（相比 500），使用更高的基础学习率 lr0 = 2 × 10−3（相比 2.08 × 10−4），并将早停 patience 设为 20（相比 25）。

7.3. 评估指标

我们使用指示函数 1[·]，定义为：如果命题 s 为真，则 1[s] = 1，否则为 0。令 G 为图像的真实框集合，P 为预测框集合，图像的像素域为 Ω。

PageIoU。

我们在像素上定义占用掩码：

MG (p) = 1[∃ g ∈ G s.t. p ∈ g]，  (2)

MP (p) = 1[∃ b ∈ P s.t. p ∈ b]。  (3)

PageIoU 衡量框并集之间的布局级重叠：

PageIoU(P, G) =
\frac{\sum_{p∈Ω} MP (p) MG (p)}
{\sum_{p∈Ω} 1[MP (p) + MG (p) > 0]}。  (4)

Label PageIoU。令 c(g) 和 c(b) 为类别标签。我们通过将每个像素分配给覆盖该像素的最小面积框的标签（否则为背景），构建逐像素标签图 LG (p) 和 LP (p)。Label PageIoU 仅在标签一致时计入交集：

LabelPageIoU(P, G) =
\frac{\sum_{p∈Ω} 1[LP (p) = LG (p) ∧ LG (p) ̸= bg]}
{\sum_{p∈Ω} 1[MP (p) + MG (p) > 0]}。  (5)

Recall@50。令 IoU(b, g) = \frac{|b∩g|}{|b∪g|}。如果存在一个预测 b 满足 IoU(b, g) ≥ 0.5（并且，对于标签感知召回，c(b) = c(g)），则真实框 g 被匹配。我们按预测置信度贪婪地计算一对一匹配。于是：

Recall@50 =
\frac{1}{|G|} \sum_{g∈G} 1[g is matched]。  (6)

PixCov（像素覆盖率）。对于具有稀疏目标标注的数据集（例如 ScreenSpot），我们报告已标注目标区域的像素覆盖率：

PixCov(P, G) =
\frac{\sum_{p∈Ω} MP (p) MG (p)}
{\sum_{p∈Ω} MG (p)}。  (7)

mAP@50。我们使用 IoU ≥ 0.5（同类别）下的一对一贪婪匹配来计算标签感知 mAP@50，在可用时按置信度对预测排序（否则使用 score = 1.0），并对真实标注中出现的类别取 AP 平均：

mAP@50 =
\frac{1}{|C^+|} \sum_{k∈C^+} AP_k @50，  (8)

其中 C^+ = {k ∈ C | |Gk | > 0} 表示在真实标注中出现的类别。

7.6. Webshot 流水线规范 7.6.1. 渲染与截图标准化 受控渲染设置。所有页面都在标准化浏览器环境（通过 Playwright 使用 Chromium）中渲染，视口尺寸为宽 1440、高 900。我们禁用了 CSS 动画/过渡，以避免运行之间出现瞬态状态和不一致的布局。我们使用有界导航超时 30s，并在提取标注之前允许一个短暂的加载后稳定期（800ms）。仅视口捕获（默认）。默认情况下，我们捕获页面顶部视口而不滚动。这使“可见 UI 元素”的定义清晰无歧义，并避免将多个滚动位置的内容混入单个样例中。支持全页面模式（通过受控滚动触发惰性加载），但为保持一致性，ScreenParse 在仅视口捕获下构建。7.6.2. 使用 DOM 层级结构进行密集 UI 元素提取 元素集合与几何。对于每个渲染页面，我们通过遍历 DOM 并保留位于视口内的元素来提取一组 UI 元素。对于每个保留的元素，我们记录：(i) 其像素坐标中的边界框，(ii) 从 HTML/ARIA 线索推断出的粗粒度元素类型（在精炼之前用作备用标签），以及 (iii) 其文本内容（来自页面文本，并带有可选的 OCR 备用；§7.6.3）。我们还记录辅助元数据，例如可访问性树快照，以供潜在的下游使用。层级结构保留。ScreenParse 的核心目标是将屏幕表示为结构化状态，而不是扁平的框集合。因此，我们保留由 DOM 诱导的父子关系：每个元素存储其父节点索引以及一个子节点索引列表（限制在提取出的可见集合内）。这产生了一种树/森林结构，用于捕获包含关系和分组（例如，包含标签页和按钮的导航栏），从而支持层级序列化（ScreenTag；§7.6.5）以及超越单元素定位的训练目标。多框架（iframe）处理。网页通常在 iframes 中嵌入内容。我们从主框架以及嵌入框架中提取元素，并将其坐标映射到共享的页面坐标系中，使所有框都可比较并可联合序列化。7.6.3. 文本提取与 OCR 细粒度文本框。除逐元素文本外，我们还可选地通过收集渲染文本的边界矩形来提取细粒度文本片段。这支持更丰富的监督和分析（例如，将布局区域与文本密度分离）。OCR 备用（可选）。在启用且可用时，我们在元素裁剪图上运行 OCR（Tesseract），以便为 DOM 文本为空的元素填补缺失或不可靠的文本（这在基于 canvas 或高度脚本化的 UI 中很常见）。OCR 被用作备用信号；当 DOM 文本存在时，仍以 DOM 文本为主。7.6.4. 过滤与样本验证 原始 DOM 提取有意保持密集，并包含噪声（布局包装器、不可见伪影、冗余重叠框）。我们应用保守过滤器，旨在在保留语义上重要的 UI 的同时提高标签质量。几何与可见性过滤。我们移除明显不适合作为训练目标的元素，包括：(i) 无效框（非正宽度/高度），(ii) 几乎完全位于视口之外或可见重叠可忽略的框，(iii) 面积 < 4 像素2 的微小伪影（除非该元素被识别为重要的交互类型），以及 (iv) 行为类似页面级包装器的过大框（我们在爬取期间将最大面积设为视口面积的 50%，并对图像区域设置特殊例外）。重复抑制。为减少冗余标注，我们使用基于 IoU 的重叠检查抑制近重复框，阈值为 0.95。当检测到重复时，我们优先保留对应于交互式/语义上有意义的元素类型的框。为在数据集层面减少冗余，我们提供一种基于哈希的近重复过滤器

7.6.5. ScreenTag 序列化

我们用一种紧凑结构来表示每个屏幕，并将其称为（ScreenTag），以作为一种高效的密集解析目标。每个元素被序列化为带有离散化位置 token 的类型化标签：<tag><x 1><y 1><x 2><y 2> text children </tag>，其中坐标按 left、top、right、bottom 顺序给出，并归一化到 0–500 的网格。我们以深度优先方式遍历层次结构，并按左上角位置对兄弟节点排序，以促进稳定的阅读顺序。

词汇表与分词。为了使结构化生成更高效，我们用 ScreenTags 的单个特殊 token 扩展分词器词汇表，包括 55 个 UI 类的开始/结束标签，以及 0–500 网格上每个坐标 bin 的离散化位置 token。这避免了将标签字符串生成为多 token 片段，减少了有效序列长度，并使生成更高效。

真值清理。在写入标签之前，我们应用额外的按类重复项清理，阈值为 IoU > 0.65，并且（对于 Text/Button/Image 等不可嵌套类）采用阈值为 0.65 的基于包含关系的重复项规则。对于类似容器的类，我们保留重复项簇中最大的框；对于原子元素，我们保留最小的框。Webshot 流水线的代码将会发布。
